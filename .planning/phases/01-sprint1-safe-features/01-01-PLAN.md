# Plan 01-01 — Surge Pricing Persistence + Vehicle-Type Pricing

## Objective
Surge multiplier currently lives in-memory and resets on every Render deploy.
Move it to `platform_config` so it survives restarts. Also wire up per-vehicle-type
pricing so Lite/Plus/Moto charge different rates.

## Context
- @backend/src/routes/admin.ts — current in-memory surge route (PUT /pricing/surge)
- @backend/src/routes/pricing.ts — fare estimation
- @backend/src/services/pricing.ts — pricing logic
- @backend/src/services/configService.ts — reads platform_config table
- @backend/src/routes/drivers.ts — toggle-online uses configService
- DB table: platform_config (id=1, has min_driver_balance, commission fields)

## Tasks

### Task 1 — Add surge_multiplier + vehicle type rates to platform_config
**Layer:** Supabase migration
**Builds on:** platform_config table already exists with id=1
**Action:**
- Apply migration:
  ```sql
  ALTER TABLE platform_config
    ADD COLUMN IF NOT EXISTS surge_multiplier NUMERIC(4,2) NOT NULL DEFAULT 1.0,
    ADD COLUMN IF NOT EXISTS base_fare_lite NUMERIC(8,2) NOT NULL DEFAULT 50.0,
    ADD COLUMN IF NOT EXISTS base_fare_plus NUMERIC(8,2) NOT NULL DEFAULT 70.0,
    ADD COLUMN IF NOT EXISTS base_fare_moto NUMERIC(8,2) NOT NULL DEFAULT 35.0,
    ADD COLUMN IF NOT EXISTS per_km_lite NUMERIC(6,2) NOT NULL DEFAULT 15.0,
    ADD COLUMN IF NOT EXISTS per_km_plus NUMERIC(6,2) NOT NULL DEFAULT 20.0,
    ADD COLUMN IF NOT EXISTS per_km_moto NUMERIC(6,2) NOT NULL DEFAULT 10.0;
  ```
**Done:** Migration applies without error; `SELECT surge_multiplier, base_fare_lite FROM platform_config WHERE id=1` returns values.

### Task 2 — Update backend: persist surge + use vehicle-type rates
**Layer:** Backend (admin.ts, pricing.ts, configService.ts)
**Builds on:** configService.getPlatformConfig() already fetches platform_config
**Action:**
1. Add new fields to configService type and SELECT query
2. In admin.ts `PUT /pricing/surge`: replace in-memory `surgeMem` with DB update to platform_config
3. In pricing.ts fare estimation: pick base_fare and per_km by vehicle_type param (default to lite rates)
4. Expose vehicle-type rates in `GET /admin/pricing` response
5. Update `PUT /admin/config` schema to accept new fields
**Done:** `GET /api/v1/pricing/estimate?vehicle_type=plus` returns higher fare than `vehicle_type=lite`. Surge set via API survives a server restart (verify against DB).

### Task 3 — Update admin Config page to show/edit new fields
**Layer:** Admin web (src/pages/Config.tsx, src/pages/Pricing.tsx)
**Builds on:** Config.tsx already has a form that PUT /admin/config; Pricing.tsx shows surge slider
**Action:**
1. Config.tsx: add input fields for base_fare_lite/plus/moto and per_km_lite/plus/moto
2. Pricing.tsx: update fare preview to use selected vehicle type (add a type toggle)
**Done:** Admin can save vehicle rates via Config page; fare preview in Pricing page updates by vehicle type.
